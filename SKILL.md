# Insert instructions below
编程风格：
1. 注释
1.1 每个模块（module）应存在于单独的源文件中，源文件名与所包含模块名相同。
1.2 每个设计文件开头应包含如下注释内容：
a) 年份及公司名称
b) 作者
c) 文件名
d) 所属项目
e) 模块名称、版本号及其描述
f) 修改纪录
1.3 必须加入详细、清晰的注释行以增强代码的可读性和可移植性。
1.4 关键逻辑、复杂逻辑、状态机、跨时钟域路径必须加注释。
1.5 注释应简洁、准确，‌避免重复代码显而易见的内容‌‌。

2. 大小写
2.1 如无特别需要，模块名和信号名一律采用小写字母。
2.2 为醒目起见，常数（`define定义）/参数（parameter定义）采用大写字母。

3. 信号命名
3.1 信号命名采用传统C语言的命名方法，即在单词之间以"_"分开，如：max_delay、data_size等等。
3.2 采用有意义的、能反映对象特征、作用和性质的单词进行信号命名，以增强程序的可读性。
3.3 为避免命名过于冗长，对较长单词的应当采用适当的缩写形式，如用"buff"代替"buffer"，"en"代替"enable"，"adr"代替"address"等。
3.4 低电平有效信号加后缀 _n（如 rst_n）‌。
3.5 输入/输出信号可加 _i/_o 后缀以明确方向‌。
3.6 禁止使用 SystemVerilog/Verilog 关键字‌作为标识符‌。
3.7 数值每隔4个用下划线隔开，如32'h0000_0000，方便阅读。

4. 参数化设计
为了源代码的可读性和可移植性起见，不要在程序中直接写特定数值，尽可能采用 `define` 语句或 `parameter` 语句定义的常数或参数。

5. 空行和空格
5.1 适当地在代码的不同部分中插入空行，避免因程序拥挤不利阅读。
5.2 在表达式中插入空格，避免代码拥挤，包括：
赋值符号两边要有空格；
双目运算符两边要有空格；
示例如下：
a  <=  b;
c  <=  a  +  b;
if (a  ==  b)
a  <=  ~a  &  c;
对于三元运算，格式示例如下:
out = cond1 ? 4'b0001 :
      cond2 ? 4'b0011 :
      cond3 ? 4'b0111 : 4'b1111;

6. 对齐和缩进
6.1 不要使用连续的空格来进行语句的对齐。
6.2 采用4个空格进行语句对齐和缩进。
6.3 各种嵌套语句尤其是if...else语句，必须严格的逐层缩进对齐。

7. 代码结构与格式
7.1 每个模块独立存放于一个文件中。
7.2 ‌每行不超过80字符‌，保持代码整洁‌。
7.3 ‌端口声明顺序优先为：时钟 → 复位 → 输入 → 输出‌，若使用 interface，优先 interface 放前或按项目模板。
7.4 ‌模块例化使用名字关联‌（而非位置关联），提高可读性与维护性‌。
7.5 ‌统一使用 begin-end‌，即使只有一条语句，便于后续扩展和调试。

8. 赋值与时序逻辑
8.1 时序逻辑‌：统一使用非阻塞赋值 <=‌，触发条件为 always @(posedge clk or negedge rst_n)‌。
8.2 严禁混用阻塞与非阻塞赋值‌于同一个always块中‌。
8.3 组合逻辑：如果是 Verilog，使用 always @(*) 来实现组合逻辑；如果是 SystemVerilog，使用 always_comb。
8.4 使用组合逻辑时，必须避免组合逻辑程序中意外锁存，方法如下：
8.4.1 补全所有条件分支：
a) if 语句必须搭配 else
b) case 语句必须包含default或列出所有可能
c) 初始化所有输出变量
d) 如使用 SystemVerilog，使用 always_comb

9. 状态机设计
9.1 默认三段式；仅在状态 <= 2 且输出全为纯组合逻辑时允许二段式。
9.2 三段式状态机设计规范如下：
a) 状态机第一段，时序逻辑，非阻塞赋值，传递寄存器的状态。
b) 状态机第二段，组合逻辑，阻塞赋值，根据当前状态和当前输入，确定下一个状态机的状态。
c) 状态机第三段，时序逻辑，非阻塞赋值，根据当前状态，某些情况下还需要根据当前输入，来确定输出信号。
9.3 状态机在频率优先且状态数适中时优先使用 onehot 编码；资源敏感场景可使用 binary/gray，并在设计说明中注明理由。

10. 模块设计
10.1 整体设计时，分解为小型、可重用的模块，以提高可读性和可测试性。
10.2 从顶层模块开始，逐步将其分解为子模块。
10.3 使用 SystemVerilog 进行编码时，合理使用 modport 让接口清晰。
10.4 模块间保持一致的命名规范。

11. 跨时钟域处理
11.1 单比特用双触发同步。
11.2 多比特优先使用握手机制进行跨时钟域处理，握手信号为 req 与 ack。
11.3 对于跨时钟传输多比特数据且需要缓存的情况，应使用异步 FIFO。

12. Fmax 优化编码规范
12.1 关键数据通路默认采用流水线思维设计；当单拍组合路径过长时，必须增加寄存器级切分逻辑。
12.2 避免在同一拍内堆叠过深的组合逻辑（多层 if/else、长算术链、长比较链）；优先拆分为中间信号并寄存。
12.3 FPGA 设计中禁止手写门控时钟（如 clk & en）；统一使用时钟使能结构（if (en)）。
12.4 高扇出控制信号（如 rst_n、en、valid、state）应评估分级寄存或局部复制，降低布线延迟。
12.5 算术逻辑应采用工具友好的写法，优先推断 DSP；存储结构应采用标准模板推断 RAM/ROM，避免非预期资源映射影响时序。
12.6 在语义允许时，优先使用并行比较或 case 结构，避免过长优先级链造成关键路径增长。
12.7 高速控制场景下，状态机输出优先寄存化，减少状态到输出的组合路径。
12.8 模块边界上的关键控制与数据路径应优先寄存化，避免上下游组合逻辑直接串接。
12.9 复位策略应兼顾时序与可测性：非必要数据通路寄存器可不加异步复位；复位策略必须在模块说明中明确。
12.10 每个关键模块应定义目标频率（TARGET_FMAX_MHZ）；布局布线后需满足 WNS >= 0 且达到目标频率，未达标必须附优化说明。

13. 常见降 Fmax 反例清单（禁止写法）
13.1 禁止在单个组合 always 块中实现过长逻辑链（多层 if/else + 算术 + 比较 + 多路选择混合）。
13.2 禁止手写门控时钟（如 clk_g = clk & en），必须使用时钟使能。
13.3 禁止跨模块直连超长组合路径（上游组合输出直接驱动下游深组合），关键边界必须寄存化。
13.4 禁止对高扇出控制信号无约束扩散（如一个 valid 直接扇出到大量子模块而无缓冲/分级）。
13.5 禁止将大位宽乘加或比较链全部放在单拍完成；必须评估并实施流水线切分。
13.6 禁止在关键路径上使用不利于资源推断的“非常规写法”，导致 DSP/RAM 未被正确推断。
13.7 禁止在性能敏感路径中使用过深优先级判断链（长串 else-if）；语义允许时应改为并行比较或 case。
13.8 禁止将状态机输出全部做组合直出并跨层使用；高速场景需优先寄存输出。
13.9 禁止无必要地给大量数据通路寄存器添加异步复位，避免复位网络成为高扇出关键路径。
13.10 禁止缺少时序约束或使用不完整约束就宣称达到目标 Fmax；无有效约束的时序结果无效。

14. Fmax 优化优先级流程（未达标时必须按顺序执行）
14.1 第一步：先确认时序分析有效性
a) 检查时钟约束、生成时钟、IO约束、false path、multicycle 是否完整且合理。
b) 若约束不完整或不合理，先修约束再谈优化；禁止在无效约束上反复改代码。
c) 必须检查 CDC 相关约束与报告一致性（如异步时钟组处理），避免将 CDC 问题误判为普通时序问题。

14.2 第二步：优先处理前3条最差路径
a) 对 worst 3 paths 逐条定位路径类型：数据通路/FSM控制/跨模块边界/复位网络。
b) 每条路径必须给出一条“可落地改动”并重新实现验证（如插入一级寄存器、并行化比较、寄存化输出）。

14.3 第三步：固定优先级优化动作
a) 优先加流水线切分长组合路径。
b) 优先做边界寄存化，切断跨模块组合串接。
c) 优先降低高扇出网络影响（控制信号分级寄存/复制）。
d) 语义允许时，将长优先级链改为 case 或并行比较结构。
e) 检查算术与存储代码风格，确保可正确推断 DSP/RAM。

14.4 第四步：复位与控制路径专项优化
a) 评估异步复位扇出，非必要数据通路寄存器改为无复位或同步复位（按项目策略）。
b) 对高速控制路径，优先采用寄存化输出与简化状态转移条件。

14.5 第五步：每轮优化后的强制回归
a) 每轮修改后必须重新执行综合与布局布线，记录 Fmax、WNS、TNS 变化。
b) 同时回归功能仿真与关键断言，禁止“时序通过但功能退化”。

14.6 收敛与停止条件
a) 达到 TARGET_FMAX_MHZ 且 WNS >= 0，视为时序达标。
b) 若连续2轮优化收益极低（例如 Fmax 提升 < 3%），必须升级为架构级调整（重划流水线/重构数据通路/调整接口时序）。

15. 低功耗
15.1 当模块处于空闲时，锁定其输入信号，防止内部逻辑无谓翻转。
15.2 优先使用时钟使能（clock enable）进行功耗控制，禁止通过组合逻辑生成门控时钟。
15.3 对高翻转率数据通路增加有效位（valid）控制，无效周期保持寄存器值不变。
15.4 状态机空闲态应避免无意义跳转，降低控制路径翻转活动。

规范性要求：
1. 按生命周期开展开发：从需求、设计、实现、验证到开发资料交付，阶段完整。
2. 需求驱动开发：先有明确、可验证的RTL需求，且形成文档，再做设计与编码；禁止“先写 RTL 再补需求”。
3. 配置管理与变更控制：需求、代码、约束文件（时序/管脚/时钟）、测试记录都要受控并可回溯，推荐使用 Git 进行管理。

测试用例：
1. 为每个RTL模块编写自检用的testbench，这些testbench应能覆盖典型用例和边缘用例。
2. 使用 SystemVerilog 断言（立即断言与并发断言）验证逻辑正确性，并用断言捕捉协议违规。
3. 覆盖率不少于 95%（至少包含代码覆盖与功能覆盖），关键断言必须全通过；关键断言列表需在验证计划中预先定义。覆盖率口径由项目验证计划统一定义（如 line/toggle/FSM/covergroup）。

输出模板（AI生成内容时必须遵循）：
1. 需求摘要
- 用 3~8 条列出模块目标、输入输出行为、时序约束、异常处理要求。
- 明确时钟/复位策略（如 `posedge clk` + `negedge rst_n`）。

2. 接口定义
- 给出完整模块端口列表，顺序为：时钟 -> 复位 -> 输入 -> 输出（若使用 interface，按项目模板）。
- 标明关键信号位宽、有效电平、握手语义（如 valid/ready 或 req/ack）。

3. 设计与时序说明
- 先说明设计思路（数据通路、控制通路、关键寄存器）。
- 明确组合逻辑与时序逻辑划分，避免锁存器。
- 若含状态机，声明采用几段式、编码方式及理由。

4. RTL实现
- 提供可综合的 Verilog/SystemVerilog 代码。
- 严格遵循本规范中的命名、缩进、注释、赋值和例化规则。
- 禁止输出与需求无关的“可选实现”分支，保持单一实现版本。

5. CDC与复位策略检查
- 列出所有跨时钟路径及处理方法（双触发/握手/异步 FIFO）。
- 说明复位释放后的稳定周期和关键寄存器初值策略。

6. SVA断言
- 至少包含：协议合法性、状态机合法性、边界条件 3 类断言。
- 区分立即断言与并发断言的使用位置，并说明每类断言目的。

7. 自检Testbench
- 提供可直接运行的自检 testbench（含时钟、复位、激励、参考模型或检查逻辑）。
- 至少覆盖：典型场景、边界场景、异常场景、复位场景。
- 给出预期通过标准（如 error 计数为 0、关键断言全通过）。

8. 覆盖率与结果
- 报告代码覆盖与功能覆盖结果；未达标项必须列出改进计划。
- 明确是否达到“覆盖率 >= 95% + 关键断言全通过”的准入标准。
- 时序结果至少包含：目标频率、实测 Fmax（post-route）、WNS/TNS、最差路径类型、下一步优化动作；并注明所用 PVT/速度等级/实现策略。

9. 交付清单
- 列出交付文件：RTL、约束、断言文件、testbench、仿真脚本、覆盖率报告、变更记录。
- 标明文件名建议与目录建议，确保可追溯与可复现。
